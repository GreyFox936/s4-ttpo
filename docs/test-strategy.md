# Стратегия тестирования F-Bank

## Объект тестирования

Объект тестирования — учебный веб-сервис F-Bank для перевода денежных средств со счёта на карту.

Сервис запускается локально как статическое веб-приложение через `python3 -m http.server 8000`.

## Основание для тестирования

Использованы требования из инструкции к итоговому заданию:

- сервис готов для осуществления рублёвых переводов;
- для перевода нужно выбрать рублёвый счёт;
- номер карты должен состоять ровно из 16 цифр;
- доступная сумма рассчитывается как `сумма на счёте - резервная сумма`;
- комиссия за перевод составляет 10% и округляется вниз;
- если сумма перевода с комиссией не превышает доступную сумму, перевод можно осуществить;
- если перевод возможен, появляется кнопка `Перевести`;
- после нажатия на кнопку появляется уведомление о принятии запроса банком.

## Цели тестирования

1. Проверить отображение основных элементов интерфейса.
2. Проверить открытие формы рублёвого перевода.
3. Проверить ввод и форматирование номера карты.
4. Проверить расчёт комиссии.
5. Проверить блокировку перевода при невалидных данных.
6. Проверить граничные условия доступной суммы.
7. Проверить согласованность интерфейса и бизнес-логики при выборе валютных счетов.

## Использованные методы и техники

### Метод чёрного ящика

Основные ручные и автоматизированные проверки выполнялись через веб-интерфейс без изменения исходного кода приложения.

### Эквивалентные классы

Для суммы перевода выделены классы:

- валидная положительная сумма;
- нулевая сумма;
- отрицательная сумма;
- сумма больше доступного остатка;
- сумма, при которой `сумма + комиссия == доступная сумма`.

Для номера карты выделены классы:

- меньше 16 цифр;
- ровно 16 цифр;
- больше 16 цифр;
- ввод с нецифровыми символами.

### Граничные значения

Проверялись границы:

- номер карты: 15 / 16 / 17 цифр;
- сумма перевода: отрицательное значение / 0 / положительное значение;
- доступный остаток: меньше, равно и больше суммы перевода с комиссией.

### Тестирование на основе опыта

Дополнительно проверены типичные риски финансовых форм:

- отрицательная сумма перевода;
- некорректное ограничение длины номера карты;
- несогласованность выбранного счёта и используемого баланса.

## Набор тестов

Тесты сгруппированы в три блока:

1. `Smoke` — базовая доступность и отображение интерфейса.
2. `Validation` — валидация номера карты и суммы.
3. `Business logic` — расчёт доступности перевода и работа со счетами.

## Автоматизация

Автоматизированные тесты написаны на Python с использованием Selenium WebDriver и Pytest.

GitHub Actions выполняет:

1. checkout репозитория;
2. установку Python-зависимостей;
3. установку Chrome/ChromeDriver;
4. запуск локального HTTP-сервера;
5. запуск Selenium-тестов.

Часть автотестов ожидаемо падает, потому что фиксирует найденные дефекты приложения.
